# ドキュメント差分チェック

ドキュメントとソースコードの乖離を検出し、LOG.md に記録する。

## Procedure

### 1. ドキュメントの読み込み

`$ARGUMENTS` に応じて対象を絞る（デフォルトは `all`）:

- `spec` または `all` → `docs/REQUIREMENTS.md` を読み込む
- `design` または `all` → `docs/DESIGN.md` を読み込む
- `task` または `all` → `docs/TASK.md` を読み込む

ドキュメントが空または未作成の場合はスキップする。

### 2. ソースコードの分析

プロジェクトのソースコード（主に `app/` 配下）を分析し、実装の実態を把握する:

- ディレクトリ構成
- データモデル、スキーマ定義
- API エンドポイントとリクエスト/レスポンス
- 画面構成、ルーティング
- 認証・認可の方式
- 外部サービスとの連携

### 3. 乖離の検出

ドキュメントの記述とソースコードの実態を照合し、以下の観点で乖離を検出する:

**仕様書（REQUIREMENTS.md）との乖離:**

- 仕様に記載されているが未実装の機能
- 仕様にない機能が実装されている
- 仕様と異なる振る舞いの実装

**設計書（DESIGN.md）との乖離:**

- 設計と異なるアーキテクチャやデータモデル
- 設計と異なる API のエンドポイントやレスポンス形式
- 設計に記載されていない技術的判断

**タスクリスト（TASK.md）との乖離:**

- 完了マーク（`[x]`）が付いているが実装が見つからない
- 実装済みだが完了マークが付いていない

### 4. LOG.md への記録

検出した乖離を `/log ai [差分] <内容>` で記録する。

例: `/log ai [差分] DESIGN.md ではJWT認証だが実装はセッション方式`

ルール:

- 内容の先頭に `[差分]` を付けて、通常の発言と区別する
- 1つの乖離につき1回 `/log ai` を実行する
- 事実のみを記述し、判断や提案は AI コメント欄に委ねる（`/log` で分析時に記入される）

### 5. 完了報告

```text
差分チェック完了
═══════════════════════════════════════
対象:     <spec / design / task / all>
検出:     <N 件の乖離>

記録済み:
  - <乖離の要約1>
  - <乖離の要約2>
  ...

乖離なし:
  - <問題がなかったドキュメント>

Next steps:
  - /log で AI コメントを付与し、対応方針を確認する
═══════════════════════════════════════
```

乖離が1件もない場合:

```text
差分チェック完了
═══════════════════════════════════════
対象:     <spec / design / task / all>
検出:     0 件

すべてのドキュメントとソースコードが一致しています。
═══════════════════════════════════════
```
