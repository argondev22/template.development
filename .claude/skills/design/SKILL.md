# Design

仕様書を分析し、実装に迷わないレベルの設計書を作成する。

## Procedure

### 1. 仕様書の確認

`docs/REQUIREMENTS.md` を読み込む:

- 内容がある場合 → 全セクションを分析し、設計に必要な情報を抽出する
- 空または不十分な場合 → ユーザーに警告する:

```text
⚠ docs/REQUIREMENTS.md が未作成または不十分です。
先に /requirements で仕様書を作成することを推奨します。
このまま続行しますか？
```

### 2. 既存の設計書を確認

`docs/DESIGN.md` を読み込む:

- 内容がある場合 → 既存の設計を把握し、更新モードで進める
- 空または見出しのみの場合 → 新規作成モードで進める

`$ARGUMENTS` が指定されている場合は、該当セクションのみを対象にする。

### 3. 設計の作成

仕様書の機能要件・非機能要件を基に設計書を作成する。

**重要: 実装者がこの設計書だけを見て迷いなくコードを書けるレベルの具体性で記述すること。**

具体的には:

- 抽象的な記述（「適切に処理する」「必要に応じて」）を避け、具体的な仕様を書く
- データ構造はフィールド名・型・制約まで定義する
- API はリクエスト/レスポンスの JSON 例を含める
- 処理フローは分岐条件と各分岐の振る舞いを明記する
- 図は Mermaid 記法で記述する

### 4. 設計書の構成

仕様書の内容に応じて、以下のセクションから必要なものを選んで記述する。
すべてのセクションが必須ではない — プロジェクトに関係ないセクションは省略する。

```markdown

## 設計概要

<!-- アーキテクチャの全体像と技術スタック -->
<!-- なぜこの構成を選んだか、トレードオフを簡潔に -->

## システム構成図

<!-- コンポーネント間の関係を Mermaid で記述 -->
```

### 5. レビューと確定

作成した設計書をユーザーに提示し、フィードバックを受ける:

- 修正依頼 → 該当セクションを更新する
- 「もっと詳しく」 → 指定セクションの具体性を上げる
- 承認 → ファイルに書き込んで完了
