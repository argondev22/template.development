# 開発ログ

開発ログを通じてフィードバックの記録と分析を行う。

## Procedure

### 引数ありの場合: つぶやきを追加

`$ARGUMENTS` が指定されている場合、ユーザーのつぶやきとして LOG.md に追記する。

1. `docs/LOG.md` を読み込む
2. テーブルの末尾に新しい行を追加する:

| 日時             | ユーザーのつぶやき  | AI コメント |
| ---------------- | ------------------- | ----------- |
| YYYY-MM-DD HH:MM | `$ARGUMENTS` の内容 |             |

- 日時は現在時刻を使用する
- AI コメント列は空のままにする（次回の引数なし実行で埋める）

追加後:

```text
📝 つぶやきを追加しました
```

### 引数なしの場合: AI コメントを返す

1. `docs/LOG.md` を読み込む
2. AI コメント列が空の行を探す
3. 空の行がない場合:

```text
✓ 未返信のつぶやきはありません。
```

4. 空の行がある場合、コメントを作成するために以下のドキュメントを読み込む:

- `docs/REQUIREMENTS.md` — 仕様
- `docs/DESIGN.md` — 設計
- `docs/TASK.md` — タスクリストと進捗

5. 各つぶやきの内容をドキュメントと照合し、AI コメントを記入する

### AI コメントの書き方

つぶやきの内容に応じて、以下のような観点でコメントする:

**タスクに関連がある場合:**

- 「タスク X.X で実装予定です」
- 「タスク X.X で対応済みです」

**仕様・設計に記載がない場合:**

- 「現在の仕様・設計に含まれていません。対応する場合は /spec や /design での更新が必要です」
- 「設計の〇〇セクションに関連しますが、この観点は未記載です」

**バグや問題の指摘の場合:**

- 「確認しました。原因は〇〇と思われます」
- 「〇〇の仕様と矛盾しています。/spec での見直しを推奨します」

**改善提案の場合:**

- 「対応する場合、/design で設計に反映した後 /task でタスク化するのが適切です」
- 「軽微な修正なので /task でタスク追加すれば対応できます」

コメントのルール:

- 簡潔に書く（1〜2 文）
- 根拠となるドキュメントやタスク番号を具体的に示す
- 次にどのスキルを使うべきかを提案する
- 感想や同意だけのコメント（「いいですね」）は避ける

### 完了報告

```text
ログを更新しました
═══════════════════════════════════════
File:      docs/LOG.md
Responded: <N 件のつぶやきにコメント>

対応が必要な項目:
  - <つぶやき内容> → <推奨スキル: /spec, /design, /task>
═══════════════════════════════════════
```
