# 開発ログ

開発ログを通じてフィードバックの記録と分析を行う。

LOG.md はオペレーターと AI の両方が発言できる。他のスキルから LOG.md に記録する場合も、必ずこのスキルを経由する。

## Procedure

### 引数の判定

`$ARGUMENTS` の先頭が `ai` で始まる場合は AI 発言モード、それ以外はオペレーター発言モードとする。

- `/log ログイン画面が重い` → 発言者: オペレーター、内容: 「ログイン画面が重い」
- `/log ai 認証方式を JWT からセッションに変更` → 発言者: AI、内容: 「認証方式を JWT からセッションに変更」
- `/log`（引数なし）→ AI コメント分析モード

### 引数ありの場合: LOG.md に発言を追加

1. `docs/LOG.md` を読み込む
2. テーブルの末尾に新しい行を追加する:

| 発言者                 | 内容         | AI コメント | 更新日時 | 作成日時         | 対応完了 |
| ---------------------- | ------------ | ----------- | -------- | ---------------- | :------: |
| オペレーター または AI | 内容テキスト |             |          | YYYY-MM-DD HH:MM |          |

- 作成日時は現在時刻を使用する
- 更新日時・対応完了・AI コメント列は空のままにする（次回の引数なし実行で埋める）

追加後:

```text
📝 ログを追加しました（発言者: <オペレーター or AI>）
```

### 引数なしの場合: AI コメントを返す

1. `docs/LOG.md` を読み込む
2. AI コメント列が空の行を探す
3. 空の行がない場合:

   ```text
   ✓ 未返信のつぶやきはありません。
   ```

4. 空の行がある場合、コメントを作成するために以下のドキュメントを読み込む:
   - `docs/REQUIREMENTS.md` — 仕様
   - `docs/DESIGN.md` — 設計
   - `docs/TASK.md` — タスクリストと進捗

5. 各つぶやきの内容をドキュメントと照合し、AI コメントを記入する
6. コメント記入した行の「更新日時」に現在時刻を設定する
7. 対応完了と判断できる行（タスクで対応済み、仕様通りで問題なし等）には「対応完了」列に `✅` を付ける

### AI コメントの書き方

つぶやきの内容に応じて、以下のような観点でコメントする:

**タスクに関連がある場合:**

- 「タスク X.X で実装予定です」
- 「タスク X.X で対応済みです」

**仕様・設計に記載がない場合:**

- 「現在の仕様・設計に含まれていません。対応する場合は /req や /design での更新が必要です」
- 「設計の〇〇セクションに関連しますが、この観点は未記載です」

**仕様・設計通りに実装されている場合（オペレーターの認識違い）:**

- 「仕様通りの動作です（REQUIREMENTS.md の〇〇セクション参照）」
- 「設計通りに実装されています（DESIGN.md の〇〇参照）。意図的な仕様です」
- 「タスク X.X で対応済みで、現在の動作は設計に沿っています」

**バグや問題の指摘の場合:**

- 「確認しました。原因は〇〇と思われます」
- 「〇〇の仕様と矛盾しています。/req での見直しを推奨します」

**改善提案の場合:**

- 「対応する場合、/design で設計に反映した後 /task でタスク化するのが適切です」
- 「軽微な修正なので /task でタスク追加すれば対応できます」

**AI 発言への対応（発言者が AI の行）:**

- 実装中の設計乖離 → 「/design で DESIGN.md を更新してください」
- 仕様との矛盾 → 「/req で REQUIREMENTS.md を見直してください」
- 軽微な差異 → 「現行の実装で問題ありません。設計書の該当箇所を /design で更新してください」

コメントのルール:

- 簡潔に書く（1〜2 文）
- 根拠となるドキュメントやタスク番号を具体的に示す
- 次にどのスキルを使うべきかを提案する
- 感想や同意だけのコメント（「いいですね」）は避ける
- オペレーターのコメントを鵜呑みにしない。仕様書・設計書と照合した結果、オペレーターの認識が誤っている場合は根拠を示して指摘する。仕様・設計通りであれば対応不要として `✅` を付けてよい

### 完了報告

```text
ログを更新しました
═══════════════════════════════════════
File:      docs/LOG.md
Responded: <N 件のつぶやきにコメント>

対応が必要な項目:
  - <内容> → <推奨スキル: /req, /design, /task>
═══════════════════════════════════════
```
