# Log

開発ログを通じてユーザーのフィードバックに AI がコメントする。

## Procedure

### 引数ありの場合: つぶやきを追加

`$ARGUMENTS` が指定されている場合、ユーザーのつぶやきとして LOG.md に追記する。

1. `docs/LOG.md` を読み込む
2. テーブルの末尾に新しい行を追加する:

| 日時             | ユーザーのつぶやき  | AI コメント |
| ---------------- | ------------------- | ----------- |
| YYYY-MM-DD HH:MM | `$ARGUMENTS` の内容 |             |

- 日時は現在時刻を使用する
- AI コメント列は空のままにする（次回の引数なし実行で埋める）

追加後:

```text
📝 つぶやきを追加しました
```

### 引数なしの場合: AI コメントを返す

1. `docs/LOG.md` を読み込む
2. AI コメント列が空の行を探す
3. 空の行がない場合:

```text
✓ 未返信のつぶやきはありません。
```

4. 空の行がある場合、コメントを作成するために以下のドキュメントを読み込む:

- `docs/REQUIREMENTS.md` — 仕様
- `docs/DESIGN.md` — 設計
- `docs/TASK.md` — タスクリストと進捗

5. 各つぶやきの内容をドキュメントと照合し、AI コメントを記入する

### AI コメントの書き方

つぶやきの内容に応じて、以下のような観点でコメントする:

**タスクに関連がある場合:**

- 「タスク X.X で実装予定です」
- 「タスク X.X で対応済みです」

**仕様・設計に記載がない場合:**

- 「現在の仕様・設計に含まれていません。追加しますか？」
- 「設計の〇〇セクションに関連しますが、この観点は未記載です。DESIGN.md に追記しますか？」

**バグや問題の指摘の場合:**

- 「確認しました。原因は〇〇と思われます。修正しますか？」
- 「〇〇の仕様と矛盾しています。仕様を見直しますか？」

**改善提案の場合:**

- 「良い指摘です。DESIGN.md の〇〇セクションに反映することを提案します」
- 「対応する場合、タスク X の後に追加するのが適切です」

コメントのルール:

- 簡潔に書く（1〜2 文）
- 根拠となるドキュメントやタスク番号を具体的に示す
- 次のアクションが明確になるようにする（「〇〇しますか？」）
- 感想や同意だけのコメント（「いいですね」）は避ける

### コメント記入後のアクション提案

AI コメントを記入した後、対応が必要なつぶやきについてユーザーに「対応しますか？」と確認する。

ユーザーが対応を希望した場合、つぶやきの影響範囲を判断して適切なドキュメントを更新する:

**軽微（実装だけで済む）:**

例: 「ボタンの位置がずれてる」「エラーメッセージがわかりにくい」

→ `docs/TASK.md` に修正タスクを追加する

**中程度（設計変更を伴う）:**

例: 「一覧画面にフィルター機能がほしい」「レスポンスにページネーション追加して」

→ `docs/DESIGN.md` の該当セクションを更新する
→ `docs/TASK.md` にタスクを追加する

**大きい（仕様変更を伴う）:**

例: 「認証方式を OAuth に変えたい」「対象ユーザーを B2B にも広げたい」

→ `docs/REQUIREMENTS.md` の該当セクションを更新する
→ `docs/DESIGN.md` の該当セクションを更新する
→ `docs/TASK.md` にタスクを追加する

ドキュメント更新時のルール:

- 既存の記述と矛盾が生じないよう、関連箇所も確認して整合性を保つ
- タスクは既存のタスクリストの適切な位置に挿入する（末尾に雑に追加しない）

```text
  - /implement で追加タスクを実装する
═══════════════════════════════════════
```
