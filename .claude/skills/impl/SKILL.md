# 実装

タスクリストに基づいて実装を進める。

## Procedure

### 1. タスクリストの確認

`docs/TASK.md` を読み込む:

- 内容がある場合 → タスクの全体像と進捗を把握する
- 空または未作成の場合 → 停止する:

```text
⚠ docs/TASK.md が未作成です。
先に /task でタスクリストを作成してください。
```

### 2. 対象タスクの決定

**`$ARGUMENTS` が指定されている場合:**

- タスク番号（例: `1.2`）またはタスク名で該当する中タスクを特定する
- 見つからない場合はユーザーに確認する

**`$ARGUMENTS` が指定されていない場合:**

- 上から順に最初の未完了の中タスク（配下に `- [ ]` が残っているもの）を対象にする
- すべて完了済みの場合:

```text
✓ すべてのタスクが完了しています。
```

対象タスクをユーザーに提示して確認する:

```text
次のタスクを実装します:
  <タスク番号> <中タスク名>
    - [ ] 小タスク1
    - [ ] 小タスク2
    ...

進めますか？
```

### 3. 作業ブランチの確認

実装を始める前に、現在のブランチを確認する:

```bash
git branch --show-current
```

- `main` にいる場合 → 対象タスクが属する**大タスク**の名前で作業ブランチを作成する:

```bash
git checkout -b feature/<大タスクに対応する短い名前>
```

例: 大タスク「2. ユーザー認証」→ `feature/user-auth`

同じ大タスク内の中タスクはすべてこのブランチで実装する。

- `main` 以外のブランチにいる場合 → そのまま作業を続ける（新たにブランチを切らない）

### 4. 設計書の参照

`docs/DESIGN.md` を読み込み、対象タスクに関連するセクションを確認する:

- データモデル、API 設計、画面設計など、実装に必要な仕様を把握する
- 設計書に記載がない部分がある場合はユーザーに確認してから実装する

### 5. 実装

対象の中タスクに含まれる小タスクを上から順に実装する。

各小タスクごとに:

1. **実装** — 設計書の仕様に従ってコードを書く
2. **動作確認** — ビルドが通ること、明らかなエラーがないことを確認する
3. **タスク更新** — `docs/TASK.md` の該当チェックボックスを `[x]` に更新する

守るべきルール:

- CLAUDE.md のコードスタイルに従う
- 秘密情報をハードコードしない
- 設計書にない機能を勝手に追加しない
- 判断に迷う場合はユーザーに確認する
- 設計書と異なる実装をした場合や、設計書に記載がない判断をした場合は、`/log ai <内容>` で LOG.md に記録する（ドキュメントは直接更新しない）。例: `/log ai 認証方式を JWT からセッションベースに変更`

### 6. 実装結果の報告

中タスク内のすべての小タスクを完了したら報告する:

```text
実装完了
═══════════════════════════════════════
Task:     <タスク番号> <中タスク名>
Status:   <N/N completed>

Changes:
  - <変更したファイル1>: <何をしたか>
  - <変更したファイル2>: <何をしたか>
═══════════════════════════════════════
```

途中でブロッカーがあり完了できなかった場合は、完了した分と残りを明示する:

```text
実装途中
═══════════════════════════════════════
Task:     <タスク番号> <中タスク名>
Status:   <M/N completed>

Completed:
  - [x] 小タスク1
  - [x] 小タスク2

Remaining:
  - [ ] 小タスク3 — <ブロッカーの理由>
═══════════════════════════════════════
```
