# タスク管理

設計書からタスクリスト全体を作成する、または個別のタスクを既存リストに追加する。

## モード判定

- **引数なし** → 全体作成モード（DESIGN.md からタスクリスト全体を作成・更新）
- **引数あり** → 個別追加モード（指定された内容を既存の TASK.md に追加）

---

## 全体作成モード（引数なし）

### 1. 設計書の確認

`docs/DESIGN.md` を読み込む:

- 内容がある場合 → 全セクションを分析し、実装すべき項目を洗い出す
- 空または不十分な場合 → ユーザーに警告する:

```text
⚠ docs/DESIGN.md が未作成または不十分です。
先に /design で設計書を作成することを推奨します。
このまま続行しますか？
```

### 2. 既存のタスクリストを確認

`docs/TASK.md` を読み込む:

- 内容がある場合 → 既存のタスクと進捗を把握し、更新モードで進める。完了済みタスク (`[x]`) は変更しない
- 空の場合 → 新規作成モードで進める

### 3. タスクの分解

設計書の各セクションから実装作業を抽出し、以下のルールで分解する:

**階層化:**

- **大タスク** — 機能単位（例: ユーザー認証）
- **中タスク** — コンポーネント単位（例: ログイン API）
- **小タスク** — 実装単位（例: POST /auth/login のバリデーション実装）

小タスクは **1つの作業セッションで完了できる粒度** を目安にする。

**テスト:**

テストコード（単体テスト、結合テスト等）は実装の一部として扱う。各中タスクに対応するテストの小タスクを含める:

```markdown
### 2.1 認証 API

- [ ] POST /auth/register のエンドポイント実装
- [ ] POST /auth/login のエンドポイント実装
- [ ] JWT トークン生成・検証ミドルウェア実装
- [ ] 認証 API のユニットテスト実装
```

テストタスクは対応する実装タスクの後に配置する。

**順序:**

上から順にやれば迷わない順序で並べる。具体的には:

1. 環境構築・初期設定
2. データモデル・DB スキーマ
3. バックエンド（API、ビジネスロジック）
4. フロントエンド（画面、コンポーネント）
5. 連携・結合（API 接続、E2E フロー）
6. テスト・品質
7. デプロイ・インフラ

同じ階層内では、依存される側を先に配置する。

### 4. タスクリストの作成

以下のフォーマットで `docs/TASK.md` を作成する:

```markdown
# タスクリスト

## 1. 大タスク名

### 1.1 中タスク名

- [ ] 小タスクの説明
- [ ] 小タスクの説明

### 1.2 中タスク名

- [ ] 小タスクの説明
- [ ] 小タスクの説明

## 2. 大タスク名

### 2.1 中タスク名

- [ ] 小タスクの説明
```

ルール:

- チェックボックス (`- [ ]`) は小タスクにのみ付ける
- 大タスク・中タスクは見出し（番号付き）で構造化する
- 各小タスクは何をすればいいか一目でわかる記述にする（「〜を実装」「〜を設定」「〜を作成」）
- 曖昧な表現（「必要に応じて対応」「適宜実装」）は使わない

### 5. レビューと確定

作成したタスクリストをユーザーに提示し、フィードバックを受ける:

- タスクの追加・削除・順序変更 → 反映する
- 粒度の調整（「もっと細かく」「まとめて」） → 該当タスクを再分解またはマージする
- 承認 → ファイルに書き込んで完了

### 6. 完了報告

```text
タスクリストを作成しました
═══════════════════════════════════════
File:     docs/TASK.md
Mode:     <新規作成 / 更新>
Tasks:    <総タスク数>
═══════════════════════════════════════
```

## 個別追加モード（引数あり）

### 1. 既存のタスクリストを確認

`docs/TASK.md` を読み込む:

- 空または未作成の場合 → 停止する:

```text
⚠ docs/TASK.md が未作成です。
先に /task（引数なし）でタスクリスト全体を作成してください。
```

### 2. 追加内容の分析

`$ARGUMENTS` の内容を分析し、以下を判断する:

- どの大タスクに属するか
- どの中タスクに属するか（新しい中タスクが必要な場合は作成する）
- 小タスクとしてどう記述するか

`docs/DESIGN.md` と `docs/REQUIREMENTS.md` も参照し、追加するタスクの文脈を把握する。

### 3. タスクの追加

既存の TASK.md の適切な位置に小タスクを挿入する:

- 該当する大タスク・中タスクが既にある → その配下に追加
- 該当する大タスクはあるが中タスクがない → 新しい中タスクを作成して追加
- 該当する大タスクがない → 新しい大タスクを作成して追加

挿入位置のルール:

- 既存のタスクの順序（依存関係）を崩さない
- 完了済みタスク (`[x]`) の前には挿入しない
- 関連するタスクの近くに配置する

### 4. 完了報告

```text
タスクを追加しました
═══════════════════════════════════════
File:     docs/TASK.md
追加先:   <大タスク番号> <大タスク名> > <中タスク番号> <中タスク名>
内容:     - [ ] <追加したタスク>
═══════════════════════════════════════
```
