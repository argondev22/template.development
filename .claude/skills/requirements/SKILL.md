# Requirements

市場調査を基に仕様書を作成する。

## Procedure

### 1. 市場調査の確認

`docs/research/` 配下のファイルをすべて読み込む:

```bash
ls docs/research/
```

- ファイルが存在する場合 → すべて読み込み、調査結果を把握する
- `.gitkeep` のみ、または空の場合 → ユーザーに警告する:

```text
⚠ docs/research/ に市場調査が見つかりません。
仕様書の品質を高めるため、先に市場調査を実施することを推奨します。
このまま続行しますか？
```

ユーザーが続行を選んだ場合のみ次に進む。

### 2. 既存の仕様書を確認

`docs/REQUIREMENTS.md` を読み込む:

- 内容がある場合 → 既存の仕様を把握し、更新モードで進める
- 空または見出しのみの場合 → 新規作成モードで進める

### 3. ユーザーへのヒアリング

`$ARGUMENTS` があればそれを出発点にする。なければ以下を順に聞く:

1. **何を作るか** — プロダクトの目的とゴール
2. **誰のためか** — ターゲットユーザー
3. **主要な機能** — 必ず実現したいこと
4. **やらないこと** — スコープ外にしたいこと

市場調査がある場合は、調査結果を踏まえて質問や提案を行う。

一度にすべてを聞くのではなく、対話的に掘り下げる。

### 4. 仕様書の作成

ヒアリング内容と市場調査を基に、以下の構成で `docs/REQUIREMENTS.md` を作成する:

```markdown
# プロダクトビジョン

<!-- 何を作るか、なぜ作るか。1〜3 文で簡潔に -->

## ターゲットユーザー

<!-- 誰のためのプロダクトか。ペルソナや利用シーンを記述 -->

## 市場調査サマリー

<!-- docs/research/ の調査結果を要約。競合、市場規模、差別化ポイントなど -->
<!-- 調査未実施の場合は「未実施」と記載 -->

## 機能要件

<!-- 優先度: Must / Should / Could で分類 -->

### Must（必須）

- [ ] 機能名 — 説明

### Should（推奨）

- [ ] 機能名 — 説明

### Could（あれば良い）

- [ ] 機能名 — 説明

## 非機能要件

<!-- パフォーマンス、セキュリティ、可用性、スケーラビリティなど -->

## 画面・フロー概要

<!-- 主要な画面やユーザーフローの概要。詳細は docs/DESIGN.md に委譲 -->

## 技術的制約

<!-- 使用技術、インフラ制約、外部サービス依存など -->

## スコープ外

<!-- 今回のリリースでは対応しないこと。スコープクリープ防止のため明記 -->

## 用語集

<!-- プロジェクト固有の用語とその定義 -->

| 用語 | 定義 |
| ---- | ---- |
```

### 5. レビューと確定

作成した仕様書をユーザーに提示し、フィードバックを受ける:

- 修正依頼 → 該当セクションを更新する
- 承認 → ファイルに書き込んで完了

### 6. 完了報告

```text

  - /design で設計に進む
  - docs/research/ に調査を追加して精度を上げる
═══════════════════════════════════════
```
